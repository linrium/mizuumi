# Tessera Usage

Mizukagami uses Tessera to run a tile-based transparency log from the Fiber
server. Local runs use Tessera's POSIX storage driver by default. Kubernetes
uses Tessera's AWS/S3 storage driver with RustFS for objects and MySQL for
sequencing.

## Start the Server

```sh
go run .
```

By default the server listens on `:3000` and stores Tessera data under `.data/tessera`.

To choose a different address or storage directory:

```sh
MIZUKAGAMI_ADDR=:3000 \
MIZUKAGAMI_LOG_DIR=.data/tessera \
go run .
```

On first start, Mizukagami creates a checkpoint signing key at `.data/tessera/.state/signer.key` and writes the matching verifier key to `.data/tessera/.state/signer.key.pub`.

## Append an Entry

Send bytes to `POST /entries`.

```sh
curl -X POST http://localhost:3000/entries \
  --data-binary 'hello tessera'
```

The response includes the assigned Tessera log index:

```json
{
  "duplicate": false,
  "index": 0,
  "published_by": "/checkpoint"
}
```

## Inspect Status

```sh
curl http://localhost:3000/tessera
```

This returns the configured log directory, signer name, current next index, and integrated tree size.

## Read Log Resources

The server exposes Tessera's read resources through the active Tessera
`LogReader`:

```sh
curl http://localhost:3000/checkpoint
curl http://localhost:3000/tile/<tile-path>
curl http://localhost:3000/entries/<bundle-path>
```

`tile-path` and `bundle-path` are paths generated by Tessera. They are not
values you choose when appending an entry.

Tessera's default tlog-tiles layout stores entry bundles under `tile/entries`, so there is no `.data/tessera/entries` directory unless a different layout is configured.

For local debugging, inspect the generated files after appending entries:

```sh
find .data/tessera/tile -type f
find .data/tessera/tile/entries -type f
```

Then request the path relative to the `tile` directory for `/tile/*`, or relative to `tile/entries` for the `/entries/*` convenience route. For example, if this file exists:

```text
.data/tessera/tile/entries/000.p/1
```

request it through the canonical tlog-tiles path:

```sh
curl http://localhost:3000/tile/entries/000.p/1
```

or through the convenience entry-bundle alias:

```sh
curl http://localhost:3000/entries/000.p/1
```

If this file exists:

```text
.data/tessera/tile/0/000.p/1
```

request it with:

```sh
curl http://localhost:3000/tile/0/000.p/1
```

Production clients should derive these paths from the latest `/checkpoint` and the tlog-tiles layout instead of discovering files from the filesystem.

## RustFS and MySQL Backend

Set `MIZUKAGAMI_STORAGE_BACKEND=aws-s3` to use Tessera's AWS/S3 backend.
For RustFS, configure a path-style S3 endpoint:

```sh
MIZUKAGAMI_STORAGE_BACKEND=aws-s3 \
MIZUKAGAMI_AWS_S3_ENDPOINT=http://rustfs-svc.mizukura.svc.cluster.local:9000 \
MIZUKAGAMI_AWS_S3_BUCKET=mizukagami \
MIZUKAGAMI_AWS_S3_ACCESS_KEY=admin \
MIZUKAGAMI_AWS_S3_SECRET_KEY=adminadmin \
MIZUKAGAMI_AWS_MYSQL_HOST=mysql.mizukagami.svc.cluster.local \
MIZUKAGAMI_AWS_MYSQL_DATABASE=tessera \
MIZUKAGAMI_AWS_MYSQL_USER=tessera \
MIZUKAGAMI_AWS_MYSQL_PASSWORD=tessera \
go run .
```

The S3 bucket must exist before Mizukagami starts. The Kubernetes Deployment
uses an init container to create the `mizukagami` bucket in RustFS.

## Operational Notes

Keep `MIZUKAGAMI_SIGNER_KEY_FILE` persistent for every storage backend. If you
delete the signer key but keep old log data, future checkpoints will use a
different signing identity and clients that trust the previous verifier key will
reject them.

The POSIX driver is intended for local development and simple deployments. The
RustFS/MySQL Kubernetes path stores Tessera log resources in object storage and
uses MySQL for sequencing.
